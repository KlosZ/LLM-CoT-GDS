## Модуль оценки генерации

Модуль позволяет:

- создавать ручные тест-кейсы;
- сохранять входные данные и результат генерации;
- запускать оценку по выбранным кейсам;
- использовать LLM-оценщика;
- считать формальные эвристики;
- получать итоговую таблицу;
- экспортировать результаты в CSV и JSON.

---

## Логика оценки генерации

Итоговая оценка рассчитывается как взвешенная сумма LLM-оценки и формальной оценки:

```text
final_score = k * llm_score + (1 - k) * heuristic_score
```

где:

- `k` - коэффициент доверия к LLM-оценке;
- по умолчанию `k = 0.7`;
- `llm_score` - нормализованная оценка сторонней модели;
- `heuristic_score` - формальная оценка по эвристикам.

---

## LLM-оценка

Сторонняя модель оценивает результат генерации по пяти критериям:

| Критерий       | Смысл                                  |
|----------------|----------------------------------------|
| `relevance`    | релевантность входному контексту       |
| `completeness` | полнота результата                     |
| `clarity`      | ясность и структурность                |
| `usefulness`   | практическая полезность                |
| `correctness`  | корректность и отсутствие противоречий |

Каждый критерий оценивается целым числом от 1 до 5.

Перед объединением с эвристиками оценка нормализуется:

```text
normalized_score = (raw_score - 1) / 4
```

То есть:

```text
1 → 0.0
2 → 0.25
3 → 0.5
4 → 0.75
5 → 1.0
```

---

## Эвристическая оценка

Формальная оценка строится на признаках, которые можно проверить без языковой модели. Каждая эвристика возвращает
структуру:

```json
{
  "name": "text_length",
  "value": 184,
  "score": 0.85,
  "weight": 1.0,
  "comment": "Длина текста находится в допустимом диапазоне."
}
```

Общий `heuristic_score` считается как взвешенное среднее:

```text
heuristic_score = sum(metric.score * metric.weight) / sum(metric.weight)
```

---

## Эвристики для итоговой темы

Для согласованной темы проверяются непустая формулировка, допустимая длина, связь с темой студента, связь с темой
преподавателя, сохранение предметной области, отсутствие явного конфликта, наличие результата разработки, наличие
метода, технологии или системы, а также отсутствие дословного копирования только одной стороны.

---

## Эвристики для уточняющих вопросов

Для уточняющих вопросов проверяются количество вопросов, отсутствие повторов, наличие вопросительного знака, связь с
исходными темами, отсутствие слишком общих формулировок, покрытие данных, методов, ограничений и ожидаемого результата,
а также отсутствие вопросов о том, что уже указано во входных данных.

---

## Эвристики для вопросов защиты

Для вопросов защиты проверяются количество вопросов, отсутствие повторов, наличие вопросов по реализации, данным,
качеству и ограничениям, связь с темой и контекстом работы, а также направленность на проверку понимания, а не простого
пересказа.

---

## Оценка многораундового согласования темы

Если согласование темы проходит в несколько раундов, система может оценивать не только финальный результат, но и сам
процесс достижения согласования.

Для каждого раунда рассчитывается своя оценка:

```text
round_score = k * llm_score + (1 - k) * heuristic_score
```

Затем считается итоговая оценка процесса:

```text
scenario_score =
    0.6 * last_round_score
  + 0.3 * average_round_score
  + 0.1 * convergence_score
```

где:

- `last_round_score` - оценка последнего раунда;
- `average_round_score` - средняя оценка всех раундов;
- `convergence_score` - скорость достижения согласованного результата.

Пример логики `convergence_score`:

```text
тема согласована за 1 раунд → 1.0
тема согласована за 3 раунда из 3 → 0.5
тема не согласована → 0.0
```

Это позволяет штрафовать ситуацию, когда темы уже близки, но система без необходимости проводит несколько раундов
уточнений.

## Создание ручного кейса оценки

В разделе **"Оценка генерации"** нужно раскрыть блок **"Создать кейс вручную"** и заполнить поля:

| Поле                    | Что вводить                                                                |
|-------------------------|----------------------------------------------------------------------------|
| `Название кейса`        | понятное название теста                                                    |
| `Описание / назначение` | что проверяет кейс                                                         |
| `Часть сценария`        | итоговая тема, уточняющие вопросы, процесс согласования или вопросы защиты |
| `Метод / этап`          | имя проверяемого метода                                                    |
| `input_json`            | входные данные генерации                                                   |
| `generated_output_json` | результат генерации                                                        |
| `expected_notes`        | ожидаемое поведение                                                        |
| `Теги через запятую`    | например `manual, topic, good`                                             |
| `Активный кейс`         | оставить включенным                                                        |

После заполнения нужно нажать кнопку **"Сохранить ручной кейс"**.

---

## Запуск оценки

В блоке **"Запуск оценки"** задаются коэффициент `k`, использование или отключение LLM-оценщика и название запуска.

Для проверки только формальных эвристик нужно снять галочку **"Использовать LLM-оценщик"**. Для полной оценки нужно
включить эту галочку. После этого нажимается кнопка **"Запустить оценку по выбранным кейсам"**.

---

## Итоговая таблица

После запуска формируется таблица результатов. В ней отображаются:

- идентификатор запуска;
- идентификатор кейса;
- название кейса;
- часть сценария;
- метод;
- оценки LLM по пяти критериям;
- нормализованный `llm_score`;
- `heuristic_score`;
- итоговый `final_score`;
- комментарий;
- подробности эвристик;
- полный JSON результата.

Результаты можно скачать в форматах CSV и JSON.
