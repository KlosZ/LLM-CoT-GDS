# LLM Lab Defense Automation — система автоматизации защиты лабораторных/практических работ с применением больших языковых моделей (LLM-LabDefAS)

Прототип Streamlit‑приложения для **анализа методических материалов**, **автоматизированной защиты студента** (вопросы → ответы → оценка), **логирования** и **калибровки через обратную связь поведения** системы с применением больших языковых моделей.

Проект реализует сценарий: *этап подготовки* (один раз, выполняется преподавателем) и *этап защиты* (много раз, выполняется студентами), плюс “дообучение” в виде **policy memory** (память предпочтений преподавателя).

---

## 1. Общее описание

Система работает в двух ролях:

- **Преподаватель**
  - загружает материалы лабораторной;
  - запускает генерацию методички/рубрики/банка вопросов;
  - проводит калибровку: помечает вопросы как good/bad и публикует версию политики;
  - просматривает логи защит и выставляет обратную связь, чтобы улучшать вопросы.

- **Студент**
  - загружает свою работу;
  - получает предварительный анализ соответствия заданию/структуре;
  - проходит мини‑защиту (вопросы разной сложности);
  - получает подтверждение, что лог отправлен на проверку;
  - оставляет feedback о процедуре.

## 2. Возможности (что умеет прототип)

### 2.1. Подготовка (роль: преподаватель)
- Загрузка материалов по типам: `goal`, `assignment`, `theory`, `report_structure`, `example`, `other`.
- Генерация комплекта:
  - методические указания по конкретной работе;
  - рубрика оценивания (критерии + веса);
  - банк вопросов (easy/medium/hard) со ссылками на источники (materials refs).
- Калибровка вопросов (TOP‑N):
  - разметка good/bad + теги причин + комментарии;
  - сохранение в policy memory;
  - публикация версии (версионирование “как спрашивать” под преподавателя).

### 2.2. Защита (роль: студент)
- Загрузка работы (pdf/docx/txt/md/ipynb/zip — зависит от парсеров).
- Предварительный анализ:
  - соответствие структуре;
  - покрытие задач;
  - важные риск‑флаги.
- Диалог защиты:
  - подбор следующего вопроса с учётом плана сложности и policy memory;
  - оценка ответа (score/label/missing_points/major_errors);
  - опциональный follow‑up вопрос (уточнение), если ответ частичный.

### 2.3. Логи, ревью, “дообучение”
- Лог защиты: работа + вопросы + ответы + оценки системы.
- Интерфейс преподавателя для ревью вопросов/ответов (good/bad + теги причин).
- Авто‑предложение обновлений policy memory по фидбеку преподавателя.
- Экспорт лога защиты в JSON.

## 3. MAS / мультиагентность (как это устроено)

В проекте используется **мультиагентная декомпозиция по ответственностям** (узкие “агенты”):

- `IngestAgent` — загрузка материалов/работ и извлечение текста;
- `MethodicsAgent` — генерация методички/рубрики/банка вопросов и подготовка к калибровке;
- `DefenseAgent` — анализ работы, подбор следующего вопроса, оценка ответа;
- `FeedbackAgent` — обработка обратной связи преподавателя и обновление policy memory.

Дополнительно предусмотрен файл `graphs.py` для **оркестрации через LangGraph** (супервизор выбирает узел по `action`).  
Streamlit‑UI (`app.py`) может работать **без LangGraph** — напрямую через `core.py`.

## 4. Хранение истории и “дообучение” (ключевая идея)

### 4.1. Почему не fine-tuning
Fine‑tuning:
- требует датасет, времени и инфраструктуры;
- усложняет демонстрацию в рамках учебного проекта.

В прототипе “дообучение” реализовано как **policy memory**:
- сохраняем **примеры удачных/неудачных вопросов** и заметки преподавателя;
- при выборе следующего вопроса подмешиваем эти примеры в промпт (good/bad few‑shot);
- система постепенно адаптируется к стилю, глубине и тематике вопросов.

### 4.2. Версионирование политики
`published_version` в лабораторной фиксирует состояние policy memory.  
Каждая защита запоминает, какая версия использовалась — удобно для воспроизводимости и аудита.

## 5. Структура проекта

```
project/
  app.py          # Streamlit UI (две роли: преподаватель/студент)
  core.py         # сценарии use-case (шаги подготовки/защиты/ревью)
  agents.py       # агенты: ingest/methodics/defense/feedback
  prompts.py      # промпты + JSON-схемы ответов LLM
  llm.py          # единая точка вызова LLM (JSON-schema ответы)
  parsers.py      # извлечение текста из файлов
  storage.py      # SQLite + файловое хранилище + экспорт лога
  graphs.py       # (опц.) LangGraph оркестрация
```

## 6. Установка

### 6.1. Требования
- Python **3.9+**
- Windows/Linux/macOS (для Windows см. раздел “Частые проблемы и решения”)

### 6.2. Установка зависимостей
Рекомендуется виртуальное окружение:

```bash
python -m venv .venv
# Windows:
.\.venv\Scripts\activate
# Linux/macOS:
source .venv/bin/activate

pip install --upgrade pip
pip install streamlit python-docx
```

Для PDF рекомендуем PyMuPDF:

```bash
pip install pymupdf
```

Опционально (если `llm.py` использует official SDK):

```bash
pip install openai
```

> `graphs.py` (LangGraph) — опционально:
> ```bash
> pip install langgraph
> ```

## 7. Настройка `.env`

Создайте файл `.env` рядом с `app.py`:

```env
OPENAI_API_KEY=ваш_ключ
MODEL_NAME=ваша_модель (прим. gpt-4o-mini)
LLM_TEMPERATURE=0.2
LLM_MAX_TOKENS=1200
LLM_TIMEOUT_SECONDS=60
# OPENAI_BASE_URL=https://api.openai.com/v1  (опционально)
```

## 8. Запуск

### 8.1. Обычный запуск
```bash
streamlit run app.py
```

### 8.2. Windows: если появляется ошибка watchdog/file watcher
Запускайте в режиме polling:

```bash
streamlit run app.py --server.fileWatcherType=poll
```

Или отключите watcher:

```bash
streamlit run app.py --server.fileWatcherType=none
```

## 9. Контрольный пример (для сдачи)

### 9.1. Материалы преподавателя
Загрузите материалы (по одному файлу) во вкладке **Преподаватель → Материалы**:
- `goal` — цель;
- `assignment` — постановка задачи;
- `theory` — теория/понятия;
- `report_structure` — структура отчёта;
- `example` — пример/шаблон.

### 9.2. Генерация
Во вкладке **Преподаватель → Генерация** нажмите:
- **Сгенерировать (методичка + рубрика + банк)**

### 9.3. Калибровка и publish
Во вкладке **Преподаватель → Калибровка**:
1) **Получить batch**  
2) отметить вопросы `good/bad`, добавить теги причин  
3) **Сохранить разметку в policy memory**  
4) **Опубликовать версию (publish)**

### 9.4. Защита студента
Во вкладке **Студент**:
1) загрузить работу и нажать **Начать защиту**  
2) нажимать **Получить следующий вопрос** → отвечать → отправлять ответ  
3) (опционально) ответить на follow‑up  
4) **Экспортировать лог защиты (JSON)**

### 9.5. Ревью преподавателя и обновление policy
Во вкладке **Преподаватель → Сессии и ревью**:
1) выбрать сессию  
2) отметить good/bad по вопросам/ответам + теги причин  
3) **Предложить и применить обновление policy**

## 10. Детали реализации (важные моменты)

### 10.1. План сложности (параметризуемо)
План можно хранить в `lab.config["question_plan"]`, например:
```json
{"easy": 3, "medium": 2, "hard": 1}
```

### 10.2. Follow-up вопросы и защита от зацикливания
Рекомендуемый параметр:
- `max_followups_per_question` (по умолчанию 1)

Идея: follow-up задаётся **только** если ответ частичный и нужно уточнение.

### 10.3. Формат логов
Экспортируемый лог включает:
- метаданные сессии;
- список turn’ов (вопрос → ответ → оценка);
- student_feedback и teacher_feedback (если есть);
- policy_version, использованную при защите.

---

## 11. Частые проблемы и решения

### 11.1. Streamlit: “session_state … cannot be modified after widget…”
Используйте `st.form(clear_on_submit=True)` для поля ответа (см. `app.py`), чтобы очищать ввод корректно.

### 11.2. Пустой текст из PDF
Если PDF — скан (картинка), без OCR текст извлечь невозможно.  
Решения:
- загрузить docx/txt;
- или добавить OCR (в текущем прототипе не включено).

### 11.3. Rate limits / ошибки LLM
Уменьшите:
- число вопросов в банке,
- `LLM_MAX_TOKENS`,
- частоту запросов,
- используйте более экономичную модель.
