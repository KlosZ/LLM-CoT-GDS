# LLM Complex Task Generation & Defense System — система формирования и защиты комплексных учебных заданий с применением больших языковых моделей (LLM-CoT-GDS)

Веб-приложение для формирования, согласования и защиты учебных заданий с использованием больших языковых моделей (далее
LLM) и эвристических сценариев.

Система поддерживает не только лабораторные работы (НИР-3), но и более широкий класс заданий: практические работы, НИР,
курсовые, доклады и другие форматы, которые можно согласовать между студентом и преподавателем.

## 1. Общее описание

Система ориентирована на учебный сценарий, в котором преподаватель и студент работают с определенным комплексным учебным
заданием (будь то лабораторная / практическая работа, курсовая или НИР). Приложение помогает автоматизировать рутинные
этапы такого без замены преподавателя:

- согласование темы;
- уточнение требований;
- формирование методических материалов;
- генерация вопросов для защиты;
- оценка полноты и качества ответов;
- фиксация результатов защиты;
- анализ качества генерации языковой модели.

## 3. Начало работы

### Требования

- Python 3.9+
- OpenAI и доступ к нему
- Наличие свободного дискового пространства

### Перейти в папку проекта

```bash
cd путь_к_папке_проекта
```

### Создать виртуальное окружение

Для Windows:

```bash
python -m venv .venv
.venv\Scripts\activate
```

Для Linux / macOS:

```bash
python3 -m venv .venv
source .venv/bin/activate
```

### Установить зависимости

```bash
pip install -r requirements.txt
```

Если файла `requirements.txt` нет, минимально потребуются:

```bash
pip install streamlit python-dotenv openai pandas
```

---

### Настроить `.env`

В корне проекта нужно создать файл `.env`.

Пример содержимого:

```env
PROXYAPI_API_KEY=ваш_ключ
OPENAI_BASE_URL=https://api.proxyapi.ru/openai/v1
MODEL_NAME=gpt-4o-mini

LLM_TEMPERATURE=0
LLM_MAX_TOKENS=6000
LLM_TIMEOUT_SECONDS=120
LLM_DOTENV_OVERRIDE=1

APP_DB_PATH=data/app.sqlite3
APP_UPLOAD_DIR=uploads

RELATION_THRESHOLD=0.55
MAX_ALIGNMENT_ROUNDS=3
MAX_DEFENSE_QUESTIONS=6
```

Пояснение переменных:

| Переменная              | Назначение                                           |
|-------------------------|------------------------------------------------------|
| `PROXYAPI_API_KEY`      | API-ключ ProxyAPI                                    |
| `OPENAI_BASE_URL`       | базовый адрес OpenAI-compatible API                  |
| `MODEL_NAME`            | модель, используемая приложением                     |
| `LLM_TEMPERATURE`       | степень вариативности ответов                        |
| `LLM_MAX_TOKENS`        | максимальный размер ответа модели                    |
| `LLM_TIMEOUT_SECONDS`   | тайм-аут запроса                                     |
| `LLM_DOTENV_OVERRIDE`   | разрешает `.env` перезаписывать переменные окружения |
| `APP_DB_PATH`           | путь к SQLite-базе                                   |
| `APP_UPLOAD_DIR`        | директория загрузок                                  |
| `RELATION_THRESHOLD`    | базовый порог согласования                           |
| `MAX_ALIGNMENT_ROUNDS`  | число раундов уточнения                              |
| `MAX_DEFENSE_QUESTIONS` | максимум вопросов на защите                          |

---

### Запустить приложение

Из корневой папки проекта выполнить:

```bash
streamlit run app.py
```

После запуска Streamlit откроет приложение в браузере. Обычно локальный адрес выглядит так:

## 4. Работа с приложением

### 1. Согласование темы

В разделе **"Согласование темы"** пользователь вводит тему студента и тему преподавателя. Система может оценить близость
тем, определить необходимость уточнения, сгенерировать уточняющие вопросы, сформировать итоговую согласованную тему и
сохранить результат для дальнейших этапов.

### 2. Методичка и публикация

Раздел предназначен для подготовки учебных материалов. Система может формировать цель работы, задачи, требования к
результату, критерии оценивания, методические пояснения и контрольные вопросы.

### 3. Работа студента

В разделе анализируется работа студента. Система может выделять сильные и слабые стороны, проверять соответствие теме и
формировать рекомендации для защиты.

### 4. Защита

Раздел используется для генерации вопросов к защите и оценки ответов студента. Вопросы должны быть связаны с темой,
материалами работы, реализацией, данными, ограничениями и качеством результата.

### 5. Ревью и policy

Раздел предназначен для фиксации обратной связи преподавателя. Система может учитывать удачные и неудачные примеры
вопросов, чтобы улучшать последующие генерации.

### Модуль оценки генерации

Отдельно стоит раздел **"Оценка генерации"** и предназначен для проверки качества генерации выбранной модели. Он нужен
разработчику, преподавателю или проверяющему, который хочет оценить, насколько хорошо модель справляется с отдельными
этапами сценария. Более подробно об оценивании можно почитать в `about_evaluation.md`

## 5. Структура хранилища

Система использует SQLite и хранит данные в нескольких логических сущностях. База данных создается автоматически при
первом запуске приложения. В базе хранятся:

- задания;
- темы;
- материалы;
- вопросы;
- результаты защиты;
- policy-элементы;
- оценочные кейсы;
- запуски оценки;
- результаты оценки.

---

## 6. Поддерживаемые файлы для обработки

- Текстовые форматы (например, `.txt` или `.md`)
- `.doc`, `.docx`
- `.pdf`
- `.pptx`
- Файлы с кодом (например, `.ipynb`)
- Архивы
- ...

Основной тестовый сценарий лучше проходить на `.txt`, потому что это самый простой и предсказуемый вариант.
Список из форматов большой, однако если конкретный parser для какого-то файла не сработал, файл все равно может быть
сохранен в систему, но извлечение текста будет пустым или частичным.

## 7. Контрольный пример (что считается успешным тестом работоспособности системы)

Контрольный пример можно считать успешным, если последовательно подтверждены следующие события:

1. создано новое задание;
2. сохранены темы студента и преподавателя;
3. загружены материалы обеих сторон;
4. появился хотя бы один раунд уточняющих вопросов;
5. ответы обеих сторон сохранились;
6. тема зафиксировалась;
7. появилась согласованная спецификация;
8. методичка сгенерировалась;
9. policy memory была заполнена;
10. submission студента был создан;
11. анализ работы выполнился;
12. защита стартовала;
13. ответы на вопросы сохранились;
14. защита завершилась;
15. student feedback был сформирован;
16. teacher feedback был сохранен;
17. policy memory обновилась по замечаниям преподавателя.

## 8. Детали текущей реализации

### Принудительный первый раунд уточнений

Даже если темы уже достаточно близки по ключевым словам и материалам, система принудительно запускает хотя бы один раунд
вопросов. Это сделано для того, чтобы:

- зафиксировать ожидания студента;
- зафиксировать требования преподавателя;
- не пропускать важный этап согласования.

### Защита без сложной внешней инфраструктуры

Система не требует отдельного сервера БД и может работать локально на SQLite.

### Работа без LLM

Если модель недоступна, проект не останавливается. Вместо функций, для работы которых нужна LLM, запускаются их
эвристические аналоги. Это удобно для локального тестирования, демонстрации интерфейса и отладки сценариев.

## 9. Ограничения

Текущая версия проекта имеет несколько естественных ограничений:

- система не является и не предполагает быть использованной как полноценная замена преподавателя;
- итоговые формулировки темы пока могут быть слишком длинными и механическими;
- оценка ответов на защите пока упрощена;
- нет сложной аутентификации и разграничения пользователей;
- workflow реализован как единое приложение, а не как распределенная система ролей;
- policy memory пока использует сравнительно простую логику накопления правил.
- LLM-оценка может меняться между запусками, так как зависит от поведения модели-судьи (для стабильной проверки
  рекомендуется сохранять результаты запусков и использовать экспорт CSV/JSON).
- эвристики (формализуемые признаки) не понимают смысл текста так глубоко, как языковая модель. Они проверяют: длину,
  количество, повторы, ключевые слова, структуру и связь с контекстом.

## 10. Быстрая диагностика проблем

### Ошибка авторизации

Проверьте:

- наличие `.env`;
- наличие и правильность пункта `PROXYAPI_API_KEY`;
- корректность переменных окружения.

Возможное решение (предложено установить по умолчанию в .env):

```env
LLM_DOTENV_OVERRIDE=1
```

### Проблема: не инициализируется LLM

Проверьте:

- наличие `.env`;
- правильность `PROXYAPI_API_KEY`;
- значение `OPENAI_BASE_URL`;
- значение `MODEL_NAME`;
- доступность сети;
- лимиты или баланс API.

Для проверки работоспособности приложения можно временно запустить оценку без LLM-оценщика. В этом режиме будут
использоваться только эвристики (== не будет обращения к внешней модели).

### Ошибка лимита

Проверьте:

- лимит запросов;
- баланс;
- доступность сети;
- доступность модели.

Возможные решения:

- повторить запуск позже;
- уменьшить количество кейсов;
- использовать более дешевую модель
- временно отключить LLM-оценщик.

### Проблема: текст файла не извлекается

Проверьте:

- поддерживается ли формат файла (чуть выше указаны или конкретнее в `parsers.py`);
- установлен ли нужный parser;
- не поврежден ли файл.

### Проблема: защита не стартует

Проверьте:

- зафиксирована ли согласованная тема;
- создан ли submission;
- выполнен ли анализ работы.

### Пустые результаты оценки

Проверьте:

- выбранные кейсы для теста (активность, корректность, ...);
- фильтр по заданиям.

Возможные решения:

- переключить область проверки на **"Все задания"**;
- включить или временно отключить фильтр **"Только активные кейсы"**;
- проверить таблицу сохраненных кейсов.
